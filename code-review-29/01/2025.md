# Code Review: Topup/Deposit System Security Audit

**Date:** January 29, 2025  
**Reviewer:** AI Assistant  
**Scope:** Deposit creation, transaction verification, and credit crediting flow

## Executive Summary

The deposit system has **strong security foundations** with on-chain verification, but there are **several critical vulnerabilities** that must be addressed before production deployment. The main concern is that clients can inject incorrect transaction data during deposit creation, though the system does verify transactions on-chain before crediting.

## Critical Security Issues

### 🔴 CRITICAL: Client Can Submit Any Transaction Hash

**Location:** `apps/server/src/index.ts:2973-3123` (`POST /api/topup/deposits`)

**Issue:** When creating a pending deposit, the server accepts a transaction hash from the client without verifying:

1. That the transaction actually exists
2. That the transaction belongs to the user (`depositorAddress`)
3. That the transaction calls the correct contract (`GAMEPOINTS_CONTRACT_ADDRESS`)
4. That the transaction parameters match the submitted data

**Attack Vector:**

```typescript
// Attacker submits:
{
  txHash: "0x...someone_elses_transaction_hash...",
  amountWei: "1000000000000000000", // 1 USDC
  depositorAddress: "attacker_address"
}
```

The server will create a pending deposit record. When `checkPendingDeposits` runs, it will:

- Verify the transaction exists on-chain ✅
- Extract `pointsMinted` from the event ✅
- **But credit the attacker** even though the transaction was initiated by someone else ❌

**Impact:** Users could claim credits for deposits made by other users by submitting their transaction hashes.

**Fix Required:**

```typescript
// In POST /api/topup/deposits endpoint, verify transaction before creating deposit:
const receipt = await checkTransactionReceipt(txHashNormalized);
if (!receipt) {
  return res.status(400).json({ error: 'Transaction not found' });
}

// Verify transaction sender matches depositorAddress
const tx = await provider.getTransaction(txHashNormalized);
if (tx.from.toLowerCase() !== resolved.address.toLowerCase()) {
  return res.status(403).json({ error: 'Transaction sender mismatch' });
}

// Verify transaction is to the correct contract
if (tx.to?.toLowerCase() !== GAMEPOINTS_CONTRACT_ADDRESS.toLowerCase()) {
  return res.status(403).json({ error: 'Invalid contract address' });
}
```

### 🔴 CRITICAL: Missing Transaction Sender Verification

**Location:** `apps/server/src/lib/topup/tx-check.ts:148-326` (`checkPendingDeposits`)

**Issue:** The `checkPendingDeposits` function verifies the transaction receipt and parses events, but **does not verify that the transaction sender (`from` address) matches the `depositorAddress`** in the deposit record.

**Attack Vector:**

1. Attacker creates deposit with `depositorAddress = attacker_address`
2. Attacker submits a valid transaction hash that belongs to `victim_address`
3. System credits attacker because deposit record says `depositorAddress = attacker_address`

**Fix Required:**

```typescript
// In checkTransactionReceipt, add sender verification:
const tx = await provider.getTransaction(txHash);
if (tx.from.toLowerCase() !== expectedDepositorAddress.toLowerCase()) {
  throw new Error('Transaction sender mismatch');
}
```

### 🟡 HIGH: Missing Amount Verification

**Location:** `apps/server/src/index.ts:3095-3107`

**Issue:** The server accepts `amountWei` from the client without verifying it matches the actual transaction amount. When crediting, the system uses `pointsMinted` from the on-chain event (which is correct), but the initial deposit record could have incorrect amounts.

**Impact:** Low - The actual crediting amount comes from on-chain events, not the submitted amount. However, this creates data inconsistency and could confuse users.

**Fix Required:** Verify transaction amount matches submitted amount (or remove amount from deposit creation since it's not used for crediting).

### 🟡 HIGH: Race Condition in Credit Application

**Location:** `apps/server/src/lib/topup/tx-check.ts:234-316`

**Issue:** The idempotency check (`wasAlreadyCredited`) reads `deposit.points_minted` BEFORE updating the deposit, but there's a window where two concurrent checks could both pass the check and both credit the user.

**Impact:** Double crediting if the same deposit is checked simultaneously by multiple processes.

**Fix Required:** Use database-level locking or unique constraint:

```sql
-- Add unique constraint on (tx_hash, points_minted) where points_minted is not null
-- Or use SELECT FOR UPDATE in the transaction
```

### 🟡 MEDIUM: Depositor Address Not Validated

**Location:** `apps/server/src/index.ts:3099`

**Issue:** The `depositorAddress` is taken from `resolved.address` (session), but there's no verification that this address actually initiated the transaction. If the session is compromised, an attacker could create deposits for other addresses.

**Impact:** Medium - Session compromise could allow creating deposits for other addresses, but credits would still only go to the session's address after verification.

**Fix Required:** Always verify `depositorAddress` matches transaction sender in `checkPendingDeposits`.

## Security Strengths ✅

1. **On-Chain Verification:** All deposits are verified against actual blockchain transactions before crediting
2. **Event Parsing:** Credits are calculated from on-chain `Deposited` events, not client-submitted amounts
3. **Idempotency Check:** Prevents double crediting (though has race condition vulnerability)
4. **Transaction Hash Validation:** Format validation for transaction hashes
5. **Session Authentication:** All endpoints require valid authenticated sessions
6. **Duplicate Prevention:** Database unique constraint on `tx_hash` prevents duplicate deposits

## Code Quality Issues

### Unused Code

- ❌ `minAmountOut` parameter is accepted but not stored in database (no column in deposits table)
- ❌ `expiresAt` is accepted but not used in any validation logic

### Missing Error Handling

- ⚠️ `checkTransactionReceipt` doesn't handle RPC rate limiting or network failures gracefully
- ⚠️ No retry logic for transient RPC failures

### Performance Concerns

- ⚠️ `checkPendingDeposits` queries up to 20 deposits and checks each one sequentially (could be parallelized)
- ⚠️ No caching of transaction receipts (could be expensive for repeated checks)

### Code Clarity

- ⚠️ Magic number: `30 * 24 * 60 * 60` (lock period) should be a named constant
- ⚠️ Complex nested conditionals in `checkPendingDeposits` make it hard to follow

## Recommendations

### Immediate Actions (Before Production)

1. ✅ **Add transaction sender verification** in deposit creation endpoint
2. ✅ **Add transaction sender verification** in `checkPendingDeposits`
3. ✅ **Fix race condition** in credit application using database locking
4. ✅ **Add transaction verification** before creating pending deposit

### Short-Term Improvements

1. Add rate limiting to `/api/topup/deposits/check` endpoint
2. Add monitoring/alerts for double credit attempts
3. Add transaction logging for audit trail
4. Parallelize deposit checks for better performance
5. Add caching for transaction receipts

### Long-Term Enhancements

1. Consider using event listeners/webhooks instead of polling
2. Add admin tools to manually verify and fix deposit records
3. Add comprehensive test coverage for edge cases
4. Document the deposit flow in detail

## Testing Recommendations

2. **Test:** Concurrent deposit checks for same transaction
   - **Expected:** Should only credit once
   - **Current:** May credit twice due to race condition

3. **Test:** Submit deposit with mismatched `depositorAddress`
   - **Expected:** Should verify transaction sender matches
